# 基础概念

数据、信息、数据模型

* 数据：原始的、未经处理的符号
* 信息：经过加工、有意义的数据
* 数据库：本质上是将数据转化为有用的信息
* **数据模型**：描述数据结构及其关系的方法

# 数据库类型

**1.关系型数据库**

基于**关系模型**理论，将数据组织成一张张二维的表 （Table），也常被称为关系 （Relation）

数据通过主键（Primary Key）和外键（Foreign Key）建立连接

主要特征：**严谨性**

它的严谨性体现在：

* **强模式（Schema-on-Write）：** 你必须在插入数据之前定义好表的结构（字段名、数据类型等）。这就像在盖房子之前，必须先画好详细的建筑图纸。这种强制性确保了数据的一致性和完整性。
* **ACID特性：** 这是关系型数据库的基石，也是它在金融、交易等对数据一致性要求极高场景中不可或缺的原因。
  * **原子性（Atomicity）：** 事务中的所有操作要么全部成功，要么全部失败，就像一个不可分割的整体。转账时，扣款和存款必须同时成功或同时失败。
  * **一致性（Consistency）：** 事务执行前后，数据库从一个一致状态转移到另一个一致状态。它遵循所有定义的规则和约束。
  * **隔离性（Isolation）：** 多个并发事务的执行互不干扰，就好像它们是串行执行的一样。避免了脏读、不可重复读、幻读等问题。
  * **持久性（Durability）：** 事务提交后，对数据的修改是永久性的，即使系统崩溃也不会丢失。

典型代表： MySQL, PostgreSQL, Oracle, SQL Server

**2.非关系型数据库（NoSQL - Not Only SQL）**

强调数据的**可用性**和**扩展性**

NoSQL数据库不再强调限制当前数据库中的存储类型，相比之下，有些数据是键值对，有些是文档，有些是图，它们可以根据需要灵活调整，而且可以无限扩展。这时不再需要预先定义所有数据的类型，而是根据物品的特点选择最适合的存储方式。

主要特征：**灵活性**

它的灵活性体现在：

* **弱模式/无模式（Schema-on-Read）：** 大多数NoSQL数据库不需要预先定义严格的模式。你可以在运行时灵活地添加新的字段，这使得它非常适合存储不断变化的数据，如用户生成内容、社交媒体数据等。
* **BASE特性：** 与ACID不同，NoSQL数据库通常遵循BASE原则，更侧重于最终一致性和高可用性。
  * **基本可用性（Basically Available）：** 系统在面临故障时，仍然能提供服务，虽然可能响应时间变长或功能受限。
  * **软状态（Soft State）：** 数据状态会随着时间推移而变化，不需要立即同步到所有副本。
  * **最终一致性（Eventually Consistent）：** 在没有新的更新写入时，经过一段时间后，所有副本的数据会达到一致状态。这是一种权衡，为了获得更高的可用性和扩展性，牺牲了即时的数据一致性。

## 关系-二维

在关系型数据库中， **“关系”本质上就是一张二维的表格** （Table）。这张表格有清晰的结构：

* **行（Row / Tuple / Record）：** 表格中的每一行代表一个独立的 **实体（Entity）**或一个**事实（Fact）** 。比如，在一个“学生”的表中，每一行就代表一个具体的学生。
* **列（Column / Attribute / Field）：** 表格中的每一列代表这个实体的 **一个属性（Attribute）** 。例如，学生的姓名、学号、年龄、专业等。

数据在逻辑上是按照这两个维度来组织的：横向的列定义了数据的种类（属性），纵向的行定义了数据的具体实例（记录）

## 键值存储（Key-Value Store）

键值存储是最简单、最基础的NoSQL数据库类型。它的数据模型极其直观：每个数据项都由一个**唯一的“键”（Key） 和与之关联的 “值”（Value**组成

类似哈希和字典，要知道“字”（Key），就能瞬间找到“解释”（Value）

## “增删改查”基础操作

### `INSERT`

`INSERT INTO `插入数据

```sql
INSERT INTO <table> (<数据明细>) VALUES
(<具体数据>),
(<具体数据>),
(<具体数据>);
```

数据明细和具体数据中的各项一一对应

### `SELECT`

SELECT 查询数据

```sql
SELECT
    <需查询数据项>
FROM
    <table>
WHERE
    <限定范围>
    salary > 5000.00;
```

### `UPDATE`

修改数据

```sql
UPDATE <table>
SET
    <数据名称>=<数据内容>;
WHERE
    <限定范围>;
```

### `DELETE`

删除数据项

```sql
DELETE FROM <table>
WHERE
    department = '市场部';
```

### 完整应用

```sql
-- 连接到MySQL服务器 (如果你已经通过命令行连接，无需再次执行)
-- mysql -u root -p

-- 创建一个新的数据库 (如果还没有，建议先创建一个独立的数据库来存放表)
CREATE DATABASE IF NOT EXISTS company_db;
USE company_db;

-- 1. 创建名为`employees`的表
CREATE TABLE employees (
    employee_id INT PRIMARY KEY AUTO_INCREMENT, -- 员工ID，主键，自动递增
    full_name VARCHAR(255) NOT NULL,            -- 姓名，非空
    department VARCHAR(100),                    -- 部门
    position VARCHAR(100),                      -- 职位
    hire_date DATE,                             -- 入职日期
    salary DECIMAL(10, 2)                       -- 薪水，总共10位，小数点后2位
);

-- 讲解：
-- `CREATE TABLE employees` 用于创建名为 employees 的新表。
-- `employee_id INT PRIMARY KEY AUTO_INCREMENT`: 定义 employee_id 为整数类型，设为主键 (PRIMARY KEY) 以确保唯一性，并设置为 AUTO_INCREMENT (自动递增)，每次插入新记录时ID会自动生成。
-- `full_name VARCHAR(255) NOT NULL`: 姓名是可变长度字符串，最大255个字符，且 NOT NULL 表示该字段不能为空。
-- `department VARCHAR(100), position VARCHAR(100), hire_date DATE, salary DECIMAL(10, 2)`: 定义其他字段及其数据类型。DECIMAL(10, 2) 适合存储货币值，因为它能精确表示小数。

-- 3. 向`employees`表中插入几条员工数据
INSERT INTO employees (full_name, department, position, hire_date, salary) VALUES
('张小明', '研发部', '软件工程师', '2022-01-15', 8500.00),
('李红', '市场部', '市场经理', '2021-03-20', 9200.50),
('王大刚', '研发部', '高级软件工程师', '2020-07-01', 12000.00),
('赵丽丽', '人事部', 'HR专员', '2023-05-10', 6000.00),
('陈伟', '市场部', '销售代表', '2023-02-01', 5000.00);

-- 讲解：
-- `INSERT INTO employees (...) VALUES (...)` 用于向 employees 表中插入新记录。
-- 括号内列出要插入数据的字段名，VALUES 后面的括号内按字段顺序提供对应的值。
-- 注意：employee_id 是 AUTO_INCREMENT，所以我们不需要在 INSERT 语句中指定它的值，数据库会自动生成。

-- 4. 查询所有薪水高于 5000 元的员工的姓名和职位。
SELECT
    full_name,
    position
FROM
    employees
WHERE
    salary > 5000.00;

-- 讲解：
-- `SELECT full_name, position`：选择要查询的字段（列）。
-- `FROM employees`：指定从哪个表查询。
-- `WHERE salary > 5000.00`：使用 WHERE 子句设置查询条件，只返回薪水大于 5000 的记录。

-- 5. 更新某个员工的职位和薪水 (将员工ID为1的张小明更新为“高级软件工程师”，薪水10000)
UPDATE employees
SET
    position = '高级软件工程师',
    salary = 10000.00
WHERE
    employee_id = 1;

-- 讲解：
-- `UPDATE employees`：指定要更新的表。
-- `SET position = '...', salary = ...`：设置要更新的字段及其新值。
-- `WHERE employee_id = 1`：通过 WHERE 子句精确指定要更新哪条记录。**务必加 WHERE 子句，否则会更新所有记录！**

-- 6. 删除某个部门的所有员工 (删除'市场部'的所有员工)
DELETE FROM employees
WHERE
    department = '市场部';

-- 讲解：
-- `DELETE FROM employees`：指定要删除记录的表。
-- `WHERE department = '市场部'`：通过 WHERE 子句指定要删除的记录。这里会删除所有 department 字段值为 '市场部' 的员工。
-- **再次强调：务必加 WHERE 子句，否则会删除表中的所有记录！**
```

## 数据处理

如何构建更健壮、更高效的数据库系统？

### 数据完整性

数据是数据库的生命线，而数据完整性就是确保这些数据健康、准确、一致的关键。它防止了无效数据被写入数据库，维护了数据之间的逻辑关系。

* **实体完整性 (Entity Integrity)：** 这是最基础的完整性，通过**主键（Primary Key）**来保证。它要求表中的每一行都必须有一个唯一的标识符，并且这个标识符不能是 `NULL`。这就像每个学生都有一个唯一的学号，而且学号不能为空。它确保了每条记录的独一无二性。
* **参照完整性 (Referential Integrity)：** 这是通过**外键（Foreign Key）**来保证的。它要求一个表中的外键值必须在它所参照的主表中存在，或者为 `NULL`。例如，你不能为一门不存在的课程分配成绩，也不能删除一个还有学生选修的课程。它维护了表与表之间的关系，防止了“孤儿数据”的产生。
* **域完整性 (Domain Integrity)：** 确保列中的数据符合预定义的格式、类型或范围。例如，年龄必须是正整数，性别只能是“男”或“女”。这通过 **数据类型** （如 `INT`, `VARCHAR`, `DATE`）和 **约束** （如 `NOT NULL`, `CHECK`, `DEFAULT`）来实现。
* **用户定义完整性 (User-Defined Integrity)：** 这是根据特定业务规则定义的完整性，可以通过 **触发器（Triggers）** 、**存储过程（Stored Procedures）或应用程序逻辑**来实现。例如，规定订单金额不能为负数，或者某个商品库存不能低于安全阈值。

### 索引（Index）

* **工作原理：** 索引通常以 B-树（或其变种）的形式存储，它包含了表中一列或多列的值，以及这些值在磁盘上对应数据行的物理位置。当你在查询时使用了索引列作为条件，数据库可以直接通过索引找到数据，而无需扫描整个表。
* **优点：** 显著提高 `SELECT` 查询的速度，尤其是在大型表上。
* **缺点：** 索引本身也需要占用存储空间；每次 `INSERT`、`UPDATE`、`DELETE` 操作时，数据库都需要维护索引，这会增加写操作的开销。因此，索引并非越多越好，需要根据查询模式和数据更新频率进行权衡。
* **何时使用索引：** 经常用于 `WHERE` 子句、`JOIN` 条件、`ORDER BY` 子句中的列。主键通常会自动创建索引。

#### 索引的类型与原理

##### **1. B-Tree（平衡树）索引**

这是**最常见、最核心**的索引类型。几乎所有主流数据库，如 MySQL 的 InnoDB 存储引擎，都默认使用 B-Tree（或其变种 B+Tree）。

B-Tree 是一种自平衡的树形数据结构。它将索引列的值按顺序存储在树的节点中。每个节点都包含键（索引列值）和指向子节点的指针。当数据库需要查找某个值时，它从根节点开始，通过比较大小，决定向左还是向右的子树继续查找，直到找到叶子节点，也就是包含了最终数据行地址的地方。


**为什么要用 B-Tree？**

* **有序性：** 因为索引值是有序存储的，B-Tree 不仅支持 **等值查询** （`=`），还非常适合 **范围查询** （`>`, `<`, `BETWEEN`）和 **排序** （`ORDER BY`）。
* **平衡性：** 无论数据如何插入或删除，B-Tree 都能保持树的平衡，确保从根节点到任意叶子节点的路径长度大致相同，从而保证了查询效率的稳定。
* **高扇出性：** 它的“扁平”结构使得一次磁盘 I/O 可以读取更多的索引数据，减少了磁盘寻道次数，从而提升性能。


##### **2. 哈希索引**

哈希索引使用**哈希函数**将索引列的值计算成一个哈希值，然后将这个哈希值和数据行的地址存储在一个哈希表中。

**优点：**

* **极快的等值查询速度：** 如果查询条件是 `=`，哈希索引的性能理论上是最好的。因为它通过哈希函数直接计算出数据行的地址，可以实现 **O**(**1**) 的时间复杂度。

**缺点：**

* **不支持范围查询：** 哈希值是无序的。你无法通过哈希索引找到一个范围内的值，比如 `WHERE age > 30`。
* **不支持排序：** 同样因为无序，哈希索引无法用于 `ORDER BY`。
* **可能存在哈希冲突：** 不同的键可能会计算出相同的哈希值，需要额外的处理来解决。

#### 创建索引

```
CREATE INDEX idx_student_name ON Students(student_name);
```

这条 SQL 语句会为 `Students` 表的 `student_name` 字段创建一个名为 `idx_student_name` 的索引。当你在 `WHERE` 子句中使用 `student_name = '张三'` 进行查询时，数据库会利用这个索引快速定位到张三的记录。

> **📝 索引的权衡**
> 索引并非没有代价。创建索引会占用额外的存储空间，并且在进行 `INSERT`、`UPDATE`、`DELETE` 操作时，数据库需要同时维护索引，这会带来额外的开销。因此，你需要权衡查询速度和写入效率，为那些经常用于查询条件的字段创建索引。

应该避免索引的场景：

* **数据量非常小的表：** 对于只有几百行的表，全表扫描可能比使用索引更快，因为查询优化器认为维护索引的开销不值得。
* **频繁更新的列：** 索引的维护是有成本的。如果一个列的数值经常被修改，数据库需要额外的时间去更新索引，这会拖慢 `UPDATE`、`INSERT`和 `DELETE`操作。
* **基数低的列：** 如前所述，像 `gender`这样的列，索引的筛选作用很弱，数据库不如直接全表扫描来得高效。
* **`WHERE`子句中不使用的列：** 如果一个列从不用于查询条件、排序或连接，那么为它建立索引就是在浪费空间和性能。

### 事务（Transaction）与 ACID

**事务**是一组**逻辑上不可分割**的数据库操作序列

它们要么全部执行成功，要么全部执行失败。

银行转账事务：

```sql
-- 开启一个事务
BEGIN TRANSACTION;

-- 步骤1：从账户A扣款
UPDATE accounts SET balance = balance - 100 WHERE account_id = 'A';

-- 步骤2：向账户B存款
UPDATE accounts SET balance = balance + 100 WHERE account_id = 'B';

-- 如果两个操作都成功，提交事务
COMMIT;

-- 如果其中任何一步失败，回滚事务，撤销所有操作
-- ROLLBACK;
```

在这个例子中，`BEGIN TRANSACTION`、`COMMIT` 和 `ROLLBACK` 构成了事务的完整生命周期。

#### ACID 特性

事务之所以可靠，是因为它必须满足 **ACID** 这四个核心特性：

* **原子性（Atomicity）：** 事务中的所有操作要么全部完成，要么全部不完成。
* **一致性（Consistency）：** 事务执行前后，数据库必须从一个一致状态转换到另一个一致状态。
* **隔离性（Isolation）：** 并发执行的多个事务互不影响，就好像它们是串行执行的一样。
* **持久性（Durability）：** 事务一旦提交，其对数据库的修改是永久性的，即使系统发生故障也不会丢失。
