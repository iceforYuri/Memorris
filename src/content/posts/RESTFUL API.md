# 互联网工作原理

> 浏览器（客户端）在向一个远方的服务器（服务端）发送请求，而服务器在接收到请求后，将相应的数据返回给浏览器

RESTful API（Representational State Transfer）正是解决这种“客户端-服务端”通信问题的一种架构风格。它不是一个具体的协议，而是一套设计原则和约束条件，旨在让分布式系统（如互联网）中的组件之间能够高效、可靠地交互。

# RESTful API的核心思想

RESTful API的设计灵感来源于Web本身，将网络上的所有事物都看作是**资源，**而其核心思想主要分为四点


* **资源** ：相当于后台的数据
  可以是任何你想要管理的东西，比如一篇文章、一个用户、一张订单，甚至是图片或视频。每个资源都有一个唯一的**URI（统一资源标识符）**来定位，就像我们用网址来定位网页一样。
* **资源的表示（Representation）** ：后台数据的转换函数
  当客户端请求一个资源时，服务器会以某种格式（例如JSON、XML、HTML）将资源的状态**表示**出来，并发送给客户端。比如，一个用户资源可能被表示成一个JSON对象，包含用户的姓名、年龄、邮箱等信息。
* **无状态（Stateless）** ：服务器不会存储任何关于客户端状态的信息。
  这是RESTful API的一个关键约束，意味着每一次请求都必须包含所有必要的信息，以便服务器能够独立地处理它。这样，服务器可以更轻松地进行扩展和负载均衡，客户端也可以更灵活地管理会话。
* **统一接口（Uniform Interface）** ：外观模式
  RESTful API使用一组统一、有限的接口来操作资源，主要就是我们熟悉的HTTP方法：
  * **GET** ：从服务器获取资源。它应该是**安全**的（不会改变资源状态）和**幂等**的（多次请求的结果是一样的）。
  * **POST** ：在服务器上创建新资源。它通常是**不幂等**的（多次提交会创建多个资源）。
  * **PUT** ：更新或替换服务器上的现有资源。它是**幂等**的（无论提交多少次，最终资源状态都是一样的）。
  * **DELETE** ：删除服务器上的资源。它也是**幂等**的。
